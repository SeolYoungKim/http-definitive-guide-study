## HTTP 메서드 : PUT, POST, PATCH
- 55페이지에서 PUT은 요청 메시지의 본문을 저장한다고 나와있고, POST 는 서버가 처리해야 할 데이터를 보낸다고 나와있습니다.
- 63페이지에서 PUT은 서버에 문서를 쓰고 이미 있다면 본문을 사용해서 교체한다고 나와있고, POST는 전송을 한다고 나와있습니다.
 
- 책의 내용만으로는 PUT을 사용하면 POST의 역할을 할 수 있는데 제가 많이 봐온 코드들은 POST로 저장을 하는 경우가 많았는데 왜 POST의 사용이 더 많은지 궁금합니다.
- 그리고 저희 회사에서는 POST로 저장하고, PUT으로 수정하며 PATCH는 사용하지 않는데 다른 분들의 POST, PUT, PATCH 의 실무 사용이 궁금합니다.

## HTTP 상태코드
- 상태코드는 크게 다섯가지로 나뉜다.
- 100-199: 정보성 상태코드 / 200-299: 성공 상태 코드 / 300-399: 리다이렉션 상태 코드 / 400-499: 클라이언트 에러 상태 코드 / 500-599: 서버 에러 상태 코드

- (56p) 만약 당신이 인식할 수 없는 상태 코드를 받게 되면, 누군가가 현재 프로토콜의 확장으로 그것을 정의했을 가능성이 있다. 
  그 상태 코드를 그것이 포함되는 범주의 일반적인 구성원으로 가정하고 다루어야한다.

- 정의된 범위에서 벗어나는 상태코드를 작성해보신적이 있으신가요? 있다면 왜 사용했는지 궁금합니다.
- 저희 회사에서는 요청에 대해 200으로 돌려보내고, 내부에서 오류가 있었으면 따로 상태코드를 본문으로 보내서 처리하고있는데 다른 분들은 어떤지 궁금합니다.
